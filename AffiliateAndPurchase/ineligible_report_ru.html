Критический баг: регекс homepage не учитывает query-параметры

Текущий регекс homepage, используемый в analysis_v2.py:226 и analysis_ineligible.py:66:

^https?://[^/]*aliexpress\.[^/]*(/(#.*)?)?$

Якорь $ в конце не пропускает URL с query-строкой. Поэтому aliexpress.ru/?gatewayAdapt=glo2rus и аналогичные URL не распознаются как главная страница — несмотря на то что CLAUDE.md прямо указывает: «query parameters do not change homepage classification».

Масштаб ошибки

В выборке топ-3000 нецелевых URL:

  • 166 594 хита с путём / в категории feed_recommendations — почти все это aliexpress.{tld}/?... главные страницы с query-параметрами
  • 17 657 хитов с путём / в search_results — аналогично: {cc}.aliexpress.com/?gatewayAdapt=…

Разрыв 12,8% (8 154 пользователя «ineligible») завышен. После исправления регекса часть этих пользователей перейдёт в категорию eligible.

Исправление

Игнорировать query-строку при проверке homepage. Правильный вариант для Python:

from urllib.parse import urlparse
import re

def is_homepage(url: str) -> bool:
    p = urlparse(url)
    return bool(
        re.match(r'^[^/]*aliexpress\.[^/?]+$', p.netloc, re.I) and
        p.path in ('', '/')
    )

Для MongoDB $regex:

"payload.url": {
    "$regex": "^https?://[^/]*aliexpress\\.[^/?]*(/(#*)?)?(\\?.*)?$"
}
После исправления необходимо перезапустить analysis_v2.py, чтобы получить скорректированный счётчик eligible-пользователей и пересчитать воронку Problem A.

Рекомендации по расширению eligible-правил

1 Исправить регекс homepage КРИТИЧНО Быстро
Текущий регекс отвергает URL вида aliexpress.ru/?gatewayAdapt=… — они не проходят $-якорь. Согласно продуктовому определению query-параметры не меняют статус главной страницы. Баг присутствует в analysis_v2.py:226 и analysis_ineligible.py:66 — а вероятно и в продакшн-коде расширения.
Парсить URL через urlparse, сравнивать только path ('' или '/'). В MongoDB — добавить (\?.*)?$ в конец регекса. Перезапустить analysis_v2.py для пересчёта воронки.
2 Активация на странице корзины ВЫСОКИЙ Быстро
316 пользователей начали сессию с /cart и позже перешли на eligible-страницу без ранней активации. Ещё важнее: в Problem B многие Purchase Completed без Purchase — это пользователи, пришедшие напрямую в корзину с прямой ссылкой, никогда не попавшие на товарную страницу.
Добавить /cart и /cart?* в список eligible-страниц для авто-редиректа. Для Firefox/Edge — webNavigation.onBeforeNavigate. Проверить, что cart не исключён в noLogUrls.
3 Парсить productId из URL страниц отзывов Лёгкое исправление
Страницы /item/{productId}/reviews имеют payload.productId = null, но product ID содержится в самом пути. ~2 785 пользователей читали отзывы — это пред-покупочное поведение.
Добавить фолбэк: при productId == null извлекать ID из re.search(r'/item/(\d+)/reviews', url). Тривиальное изменение в парсере событий.
4 A/B тест: активация на страницах поиска Стоит тестировать
4 186 пользователей C2 начали с поиска. Пути /w/{query} — активный шоппинг с высоким intent. Для Firefox/Edge авто-редирект не требует UI-изменений.
Запустить A/B тест расширения eligible-правил на поисковые страницы только для Firefox/Edge авто-редиректа. Контрольная группа — текущие правила.
5 Страницы магазина продавца Стоит тестировать
327 пользователей C2 начали с /store/{id} — средняя воронка покупки (изучение конкретного продавца). 82,2% из них всё-таки дошли до eligible-страницы.
Добавить /store/{id} в eligible-правила. Оценить конверсию в Affiliate Click до и после.
6 Страницы категорий Стоит тестировать
278 пользователей начали с /category/{id}/…, но только 36,7% дошли до eligible — самая низкая конверсия. Активация на входе в категорию поможет захватить пользователей до того, как они уйдут.
Добавить /category/{id}/… в eligible. Мониторить, не увеличивает ли это число ложных активаций у пользователей, покидающих AliExpress без покупки.

Ограничения данных

Выборка топ-3000 URL

Таксономия построена по топ-3000 URL по числу хитов. Длинный хвост редких нецелевых страниц не учтён. Счётчики пользователей по категориям приблизительны.

Двойной счёт пользователей

Один пользователь может посещать несколько категорий — поэтому суммы «пользователей по категориям» превышают общее число ineligible-пользователей (8 154). Процент от raw также может суммироваться сверх 100%.

noLogUrls

Некоторые страницы AliExpress (оформление заказа, платёж) могут быть исключены из логирования конфигом расширения. Отсутствие событий checkout/payment не означает, что пользователь там не был.

Баг с регексом homepage (§ Критический баг)

Число eligible-пользователей в analysis_v2.py занижено. Выводы о доле ineligible (12,8%) следует пересмотреть после исправления регекса и перезапуска анализа.

Первое событие в окне

«Первая страница» — первое событие в окне 2026-03-06 — 2026-04-02, а не абсолютно первое взаимодействие пользователя. Пользователи, активные до начала окна, могут иметь смещённую «первую» страницу.

CIS-наблюдаемость

Для CIS-пользователей events не содержат utm-параметры. Прямая наблюдаемость аффилейт-возврата отсутствует. Выводы о возврате для CIS основаны только на проксиметриках.
